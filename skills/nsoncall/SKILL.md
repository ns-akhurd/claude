# NS DLP Oncall Quick Reference

Static playbook distilled from ~2000 `#dlp-oncall` Slack messages (2025-04 through 2026-06). Everything below is baked in — no DB or Slack read required for routine triage. For anything not covered here, fall back to the `dlp-oncall-tracker` skill (live DB query) or search Slack directly.

## How to run a query

Use the `eng-skills:sumo` skill's CLI directly (do not guess the path — resolve it):
```bash
~/.claude/plugins/cache/netskope/eng-skills/*/skills/sumo/scripts/sumo search '<PREFIX> <your filter>' --from -1h --limit 100
```
Add `--records` for any query using `count`/`count_distinct`/`by`-grouping. Always scope with a `_sourceCategory`/`_index` prefix below before widening the time range — unscoped queries burn cost and can return zero hits even over multi-TB windows. **Always append `!(datacenter=qa)` to every query** to drop QA/synthetic traffic, unless you are specifically debugging QA — real oncall incidents are never in `datacenter=qa`.

**`_index` (chargeable debug) queries — last resort only.** MUST NOT use an `_index="dlp_*_debug"` query unless BOTH are true: (1) you already ran a `_sourceCategory`/`_source`-scoped query on the **continuous (free) partition** for the same question and it returned zero because the line you need (e.g. `file_size=`/`file_type=`/`Scan start`/`ci_time=`) is not in the continuous partition; AND (2) you **cannot proceed to a conclusion without that line** — the answer is required to triage, not nice-to-have. NEVER use `_index` for broad/category scans, host counts, tenant attribution, trend-over-time, or cross-host/cross-tenant spread (those are continuous-partition work — `_sourceHost=*dlppool*<pop>* "tenantid=<id>" \| count by _sourceHost` etc.). When you do escalate to `_index`: pair with `_sourceHost=<host>` + a specific `tenantid=`/`trid=`/`rqid=` filter + a tight `--from` window (e.g. `-90m`); run **one** scoped `_index` query to recover the specific line you need, then move back to continuous for all cross-host/cross-tenant attribution. State explicitly in your response each time you escalate to `_index`, naming the continuous query that came back empty and why the line is required — never run a second `_index` query if the first already answered.

**Report every query you ran, verbatim.** Any time you execute a live Sumo search during a triage (not just quoting the catalog), your final response MUST include the exact query string(s) and the `--from`/`--records`/other flags actually used — in a fenced code block, one per query, next to the finding it produced. Never summarize "I found X" without the query that found it. This lets the on-call engineer copy-paste and re-run it themselves rather than trust the conclusion blind. Applies to every triage response, not just ones the user flags as needing verification.

**Dump the full triage to a file, every time.** In addition to the in-chat response (never instead of it), write the complete triage record to a markdown file in `/tmp`: the alert text as given, every query executed verbatim with its flags, the key result rows/timeline, tenant/root-cause attribution, and the conclusion + recommended action. Name it `/tmp/nsoncall-<pop-or-alert-slug>-<timestamp>.md`, timestamp via `date -u +%Y%m%dT%H%MZ` (e.g. `/tmp/nsoncall-de-muc1-20260703T0855Z.md`). This gives the on-call engineer a standalone file to hand off, paste into a ticket, or diff against a later re-check — don't rely on the chat transcript being the only copy.

## Resolving the dataplane repo path

The code-grounded parts of this skill (Deep-Dive rqid technique, `IncidentReport.cpp` reference, playbook #13) need a local checkout of the `dataplane` repo. Never hardcode a path — it's different per machine/user. Resolve it fresh:

1. **Check memory first.** Search your memory system for a previously-recorded dataplane checkout path (this skill's own history writes one to a `reference-*.md` memory file when it finds one — look for it there before searching disk).
2. **Verify before trusting it** — a memory hit is a claim, not a fact: `test -d <path>/.git && git -C <path> remote get-url origin`. It must be a `dataplane` (or `dataplane2`) checkout with remote `netSkope/dataplane.git`. If the path is gone or the remote doesn't match, treat memory as stale and continue to step 3.
3. **No usable memory hit** → search common locations, e.g. `find /root/code /home/*/code ~/code ~ -maxdepth 3 -iname "dataplane*" -type d 2>/dev/null`, and verify each candidate the same way as step 2.
4. **Still nothing** → ask the user once for the path, then write it to memory (type `reference`) so future sessions don't re-ask.
5. **Path resolved** → check it's current before trusting "the code says X": `git -C <path> fetch origin develop && git -C <path> log --oneline HEAD..origin/develop`. If behind, `git -C <path> pull --ff-only origin develop` (and `git submodule update --init --recursive` if `git status` shows submodule pointer diffs afterward — check `git status --porcelain` first, per standard practice, before assuming submodules need touching).
6. **No valid path found at all** → say so explicitly and continue with the Sumo-only catalog/playbook below; don't guess at file:line citations you haven't actually read.

**HAProxy saved-search regex is per-POP brittle** (live-confirmed 2026-07-06, DE-FRA1): the catalog regex above assumes `-0..` literal anchoring right after the server-pool name (`(?<servers>[-_a-zA-Z0-9.\/]+)-0.*`). FR-A1/DE-FRA1 use `dlppool1-N-0` naming that this anchor doesn't match, so the query silently returns zero rows instead of erroring — looks like "all healthy" when it's really "query didn't match." For an ad-hoc check on a POP you haven't verified against, use a looser pattern that drops the `-0..` anchor entirely, e.g. `\S+ .*?(?<activehosts>\d+) active and`, before trusting a zero result. The catalog's exact string above is kept as-is since it's the literal production saved-search, not because the anchor is universally correct.

**HAProxy log partitions return CCR-compressed `_raw`** (confirmed DE-FRA1): the default Sumo projection shows `<<ccr:...>>` pointers instead of actual content on these partitions — append `--json` to the query to decode `_raw`.

**Common idioms** (append to any prefix above):
- `"<error string>" | count_distinct(_sourceHost) as unique_count` — how many hosts affected
- `"<error string>" | count by tenantid` — per-tenant breakdown
- `_code=<id>` links from old saved searches are **ephemeral permalinks tied to a fixed time range** — not reusable query syntax, don't copy them as templates.

**`_sourceCategory` naming convention** (live-confirmed 2026-06-29, fills a prior "not discoverable in-repo" gap): `<country>_<pop>_<mp|dp>_<component>`, e.g. `uk_lon3_mp_nsdlp_svr` (MP DLP, UK-LON3), `au_bne1_dp_nsdlp_svr` (DP DLP, AU-BNE1), `br_sao4_dp_mtnsproxy` (nsproxy, BR-SAO4). The alert-facing "Pop: UK-LON3" name does not always equal the internal datacenter tag used in `datacenter=<dc>` — cross-check both before concluding a scope returned zero hits because the alert is stale rather than because the tag guess was wrong.

**Not every POP has local DLP** (live-confirmed 2026-07-03, BR-SAO4): some POPs are edge/gateway-only — `_sourceCategory=*<pop>*` returns categories like `_dp_mtnsproxy`, `_dp_edge-nsproxy`, `_dp_chainproxy`, `_dp_nsgw` but **no** `_dp_nsdlp_svr` / `_mp_nsdlp_svr` at all. Before assuming a DLP-scoped query returned zero hits because the alert is stale, run `_sourceCategory=*<pop>*` with no other filter to see what components that POP actually has — if there's no `nsdlp_svr`, the POP forwards scans to a remote DLP backend and you need the proxy-side category (see playbook #13) instead.

**Some DLP nodes are Kubernetes pods whose `_sourceCategory` doesn't encode the POP name at all** (live-confirmed 2026-07-03, DE-MUC1): instead of `de_muc1_dp_nsdlp_svr`, the actual category was `pod:dlp-core-0__container:dlp-core` — a `_sourceCategory=*muc1*` glob returns zero for these even though the POP has active DLP traffic, because "muc1" only appears in `_sourceHost` (`DE-MUC1_knode05.c1.muc1.nskope.net`) and the `datacenter` field, never in `_sourceCategory`. **If a `_sourceCategory=*<pop>*` glob returns zero, don't conclude "no local DLP" (playbook #13) until you've also tried filtering by the native `datacenter=<pop>` field directly** (works independent of `_sourceCategory` naming, e.g. `(_sourceCategory=*dlp*) "IncidentReport.cpp" datacenter=<pop>` — this is the same `datacenter` dimension the loss-detector query already groups by).

**Some datacenters have no local IMS/LBaaS endpoint and route through a shared regional hub** (live-confirmed 2026-07-03): DE-MUC1's `IncidentReport.cpp` posts all go to `imsapisvclbaasvip-int.am2.nskope.net:8000` — AM2's endpoint, not a MUC1-local one. A resp_400/resp_504/timeout spike attributed to one datacenter's tenants can actually be a problem at a *different* hub datacenter's LBaaS/IMS tier. Check blast radius across other datacenters sharing that same hub hostname before assuming the issue is local to the alerting POP: `(_sourceCategory=*dlp*) "ERROR" "IncidentReport.cpp" "file" "<error string>" "<hub-hostname-fragment>" !(datacenter=qa) | parse "tenantid=* user=" as tenant | count as num_errors by datacenter | sort by num_errors desc` — if multiple datacenters show up, escalate the hub, not any one POP.

**Confirmed per-POP `datacenter=` value traps** (live-confirmed 2026-07-06/07 — re-verify per POP, don't assume from the alert/POP name): BR-SAO1 → `datacenter=br_sao1` returns **zero** hits, must scope via `_sourceCategory=br_sao1_dp_nsdlp_svr` instead; FR-MRS1 → the real value is `datacenter=mrs1` (not `fr_mrs1`); SG-SIN2 → the real value is `datacenter=sin2` (not `sg_sin2`). Always run one component-discovery query (`_sourceCategory=*<pop-fragment>*` with no other filter) to see the live value before trusting a `datacenter=<alert-name>` guess.

**`IncidentReport.cpp` `error=` code reference** (from the dataplane repo — see "Resolving the dataplane repo path" above — + live examples; endpoint is `'DLP Event Generator'`, `'API Connector from pop-list'`, or `'LBaaS'` depending on datacenter/routing — `'LBaaS'` posts to `imsapisvclbaasvip.<dc>.nc1.<dc>.nsscloud.net:8000`):

**`final=false` vs `final=true` matters for alert triage**: retry-count alerts (e.g. `"IMS responded with Status: 400 > N"`) count *every* attempt, including ones still retrying (`final=false`) — the DLP service hasn't given up yet, so this is not necessarily data loss. Only `final=true` rows are confirmed loss (use the loss-detector query above, which filters to `final=true`, to check whether it's escalated that far). If a resp_400/timeout/connect-failure alert is firing but the loss-detector query for the same window is empty, the retries are still absorbing it — worth confirming which specific tenant(s)/host(s) are driving the count (`| count by tenant`) and whether it's concentrated on one downstream endpoint (LBaaS-side issue, escalate) vs one tenant's oversized payloads (content issue).

**Check the trend, not just the current count**: the same `"IMS responded with Status: 400 > N"` alert text can mean two very different things. Run `| timeslice 30m | count as num by _timeslice | sort by _timeslice` over the last 3-4h before concluding either way:
- **Decaying/already at zero** (e.g. LON3 2026-07-03: peaked 342/30min at 03:00 UTC, zero by 04:00) → transient, likely already mitigated, no action needed beyond noting the driving tenant for pattern-tracking.
- **Steadily climbing** (e.g. MUC1 2026-07-03: 1 → 7 → 25 → 39 → 68 → 69 → 151 → 161 → 318 across nine 30-min buckets, roughly doubling hourly, 14+ distinct tenants) → active and worsening, escalate now per the IMF/EIMF threshold rule — don't wait for `final=true` losses to appear before escalating an escalating trend.

**Retry schedule:** the direct IMS post path retries with exponential backoff `retryInterval × 2^(retryCount-1)`, base 30s (code-verified `develop`@`libs/dlp/svc/src/IncidentReport.cpp`, `DLP_INCIDENT_RETRY_INTERVAL=30000`; LBaaS endpoint gets `m_lbaasMaxAttempts` attempts, then the API Connector SVCLB endpoint gets the remainder). Prod config (per the IMS post retry spec; see `triage-dlp-incident-post` playbook §1) sets 6 attempts over ~15.5 min — 3 against the LBaaS VIP (`endpoint='LBaaS'`), then 3 against the API Connector SVCLB (`endpoint='API Connector from pop-list'`); attempts land at t=0, +30s, +90s, +210s, +450s, +930s. `try_count=N` in the URL = attempts-so-far (0-indexed). `num_retries` on the metric = the recorded attempt's retry index, 0..5; `num_retries=5, status=fail` = all 6 attempts failed → GEF fallback fires. Some DCs override via `ims-apisvc-service.json`. The log line carries `endpoint='<LBaaS|API Connector from pop-list>'` and `final=true|false`; add `endpoint=` to the loss-detector query's parse if you need per-endpoint attribution.

**GEF fallback is incident-metadata-only.** When all 6 direct attempts fail, the incident record lands via GEF (Kafka → `ims-incident-cg` consumer → ims-apisvc `/incident`), but the forensic / original-file / preview / subfile payloads are **permanently lost** — the incident surfaces in IMS with `forensic_status="Upload Failure"`. By design, not a bug. So a "GEF fallback spike" without forensic loss is recoverable; with forensic loss it's metadata-only recovery, not full recovery.

**`home_pop` vs `pop` vs `ims_server`** (cross-pop confusion): in the incident metrics, `pop` = the DLP host's pop (what the loss-detector query groups by as `datacenter`), `home_pop` = the tenant's MP, `ims_server` = the IMS VIP in the URL (usually the tenant's `home_pop`, sometimes a failover). A tenant whose `home_pop=US-SJC1` can have incidents originating from DLP in any pop — so a resp_400/timeout spike attributed to POP X's tenants may actually be a problem at the `ims_server` VIP's POP (see the shared-regional-hub note above). When the driving tenants' `home_pop` ≠ the alerting `pop`, escalate the `ims_server` POP, not the alerting one.

## Deep-Dive Technique: Trace a Single Request via rqid / tenantid

A category-scope query (catalog above) returns many hits. To go from "a category of errors" to "the exact request and code path that failed": request-scoped log lines from services sharing the datapath logging infra (nsproxy, nsdlp, cache_svc, fplookupsvc, fpuploadsvc, watchdog, etc.) are auto-prefixed with `trid=<id> rqid=<id> tenantid=<id> user='<email>'` — no special logging needed to get it (verified: `libs/base/src/cpp/log.cpp`, format string `"trid=%zu rqid=%zu tenantid=%ld user='%s' %s\n"`; full field reference at `ai-agent/docs/handbooks/logs.md` in the dataplane repo). A small "short log" code path (e.g. Traffic Uploader) intentionally omits this prefix — if a line has no `rqid=`/`tenantid=` at all, it's one of those, not a gap in the technique.

**0. Starting from tenantid (when the alert gives you a tenant, not yet a request):** Most tenant-scoped alerts (`Profile Load Failures Per Tenant`, `DLP Scans Timeout By Pop,Tenant`, playbook #2/#4/#10) name a `tenantid` before you have any specific request. Search `<category scope> "tenantid=<id>"` first — `| count by tenantid` (Common idioms above) to confirm you have the right one — to pull every request for that tenant in the window, then take one hit's `rqid` into step 1. `tenantid` can be `0` if the tenant hasn't been identified yet at the point a line was logged (same early-connection caveat as the rqid placeholder below) — fall back to `user=<email>` as a filter in that case. Once the tenant is pinned down, its on-disk config lives at `/opt/ns/tenant/<tenantid>/cfg/` on the affected host (e.g. `netskope.nsdlp.cfg` for feature-flag state, profile lists) — cross-check that against what the logs show for a stale/mis-synced config (this path is how playbook #2's tenant-config checks were done historically).

1. From a category- or tenant-scope hit, copy its `rqid=<value>` (keep `tenantid=<value>` alongside it too — cheap insurance, since rqid derivation could theoretically collide across tenants).
2. Re-search the same time window for just that ID, dropping the original error-string filter: `sumo search '<sourceCategory/_source scope> "rqid=<value>"' --from <window> --raw`. Widen the scope across every `_sourceCategory`/`_source` the pipeline touches (nsproxy → nsdlp → cache_svc/fplookupsvc/fpuploadsvc → DSE/FP-gateway) to pull every line for that one request, not just the failing one — read through all of it for errors/warnings, not just the line that first surfaced the category.
3. Order by `_messageTime` to reconstruct the request's timeline. The "All Scan Info" line (catalog scope 1) carries per-stage timings (`fextract_time=`, `ci_time=`, `drm_time=`, etc.) — use it to pinpoint which stage actually failed or was slow.
4. **trid vs rqid**: `trid` is per-connection (shared by every request on a keep-alive connection); `rqid` is per-request and is what correlates across services. Early lines for a request (pre-L7-parsing) may show a placeholder rqid (`trid + 1`) before `HttpRequestEngine.cpp`'s `setTrxnID()` assigns the real one — if an rqid search comes up short right at connection start, also try `trid=<value>`.
5. **Known gap**: no rqid/trid propagation was found in the text-extraction service (`apps/nstextsvc`, `pkg/textsvc`) as of `develop`@`3ef2ad19e3` (grep-verified 2026-07-03, 0 matches — re-verify if the repo has since diverged far from that commit) — if the failure is inside TE, a pure-rqid search may go quiet at that hop; correlate via `trid` + `tenantid` + timestamp window instead.
6. Once the errors/log lines for that request are in hand, locate the matching code in the dataplane repo (resolve the path per "Resolving the dataplane repo path" above; don't assume a fixed location), branch `develop`, to explore further — grep for the exact log string from the failing line to land on the responsible function. Known entry points (paths relative to repo root): `libs/dlp/svc/src/ns_dlp_svc.cpp` (main scan flow, All Scan Info emission), `libs/watchdog/src/Watchdog.cpp` (WD abort logging), `apps/dlp/cache_svc/src/nsdlpcachesvc.cpp`, `apps/dlp/fplookupsvc/.../FingerprintController.cpp`, `apps/dlp/fpuploadsvc/.../UploadController.cpp` (FP/cache hops), `libs/dlp/svc/src/DseClient.cpp` / `libs/dlp/components/src/ExtFingerprinting2.cpp` (outbound DSE/FP-gateway calls; both carry `x-netskope-tenantid/trid/rqid` headers), `libs/httpdispatch/src/cpp/DlpModule.cpp` (nsproxy's DLP client — "ASP client for DLP scan timed out", playbook #13).

## Pivoting between playbooks

Mid-triage, evidence may show the real locus is a different playbook than the one you started on. Switch when a pivot trigger fires — don't keep drilling the wrong surface.

- **#9/#11 (forensics/incident loss) → #10/#2 (scan):** the DLP forensics buffer is at its 8 GB cap AND scan-stage latencies / `RESP_TIMEOUT` are elevated. Scan-path slowness inflates the incident buffer; fixing the scan path drains it. Treating it as an incident-post issue is wasted effort.
- **#10 (scan capacity/timeouts) → #9 (incident post):** `failed_scans` is flat but forensics-loss % spikes, `IncidentReport.cpp` `final=true` rows appear, or `ims-incident-cg` Kafka lag climbs. The scan worked; the post failed.
- **#13 (edge-POP scan timeout) → #10/#12:** timeouts spread across many tenants (not one dominant) → remote DLP backend/network path is degraded, not a tenant burst → escalate the remote DLP POP's capacity/backend health. If one tenant dominates → #4 (entity/regex) or #10 (tenant burst).
- **#14 (EG) → #9 (DLP-side incident post):** EG logs show RECOVERED lifecycle (no `incident drop` / LOST lines) but the `IncidentReport.cpp` `final=true` loss-detector query is hot. The DLP service owns that path, not the EG k8s service.
- **#14 (EG) → #10/#2 (scan):** EG Redis stream backing up (EG logs show stream-depth/post-latency warnings, EG posts slow) but EG itself is healthy → upstream scans are slow/failing; fixing the scan path drains the EG stream.
- **Scan failures + OCR timeouts correlating → stay on scan:** OCR is best-effort in the DLP scan pipeline; OCR timeouts do NOT fail scans. Correlation ≠ causation. (No dedicated OCR playbook row yet — treat OCR as a signal to ignore unless image-attachment OCR is the reported symptom; full OCR triage in `triage-ocr` / `knowledge/playbooks/ocr.md`.)
- **#10 (pop-wide) vs SMTP/email-only:** if all access methods show elevated `RESP_*` → pop-wide DLP (#10). If only `access_method="SMTP Proxy"` / `emailmpdlp*` / `dlp_be_email` → email path has its own saturation pattern (content-driven: one big email pins all 4 backends, `tesvc_textlen` 10–100× baseline; vs pool capacity: sustained CPU >50% under normal load). Full SMTP triage in `triage-smtp-dlp` / `knowledge/playbooks/smtp-dlp.md`.

## Surface deep-dives (precision classifiers + log patterns)

Per-surface precision distilled from the `dlp-eng` triage skills' playbooks (`knowledge/playbooks/{dlp-scan,dlp-incident-post,dlp-eg,smtp-dlp,ocr}.md`), stripped of the Prism/PromQL layer (no access). Use these to classify once you've picked a playbook row; they add the differential diagnosis and log-pattern fingerprints the flat table omits.

### Scan surface (rows 1–8, 10, 12, 13)

**5-pattern classifier (apply before drilling — Sumo-grounded):**

| Pattern | Sumo signal | → Row |
|---|---|---|
| Tenant-scoped, errors on **every host** in pop for that tenant + profile-load strings | `(_sourceCategory=*nsdlp_svr OR _source=*dlp-core*) "tenantid=<id>" ("Profile" OR "Failed loading" OR "config") \| count by _sourceHost` → every host non-zero | #2 (broken profile) |
| 100% fail on **one (host, tenant)** pair, peers clean | same query → one `_sourceHost` non-zero, rest zero for that tenant | #2 (stale-on-host variant) |
| Many tenants on **one host**, short window | `"<error>" \| count by _sourceHost, tenantid` → one host dominates | #5/#6 (host transient) |
| **All hosts** in pop elevated simultaneously | `\| count by _sourceHost` → every host non-zero | #10 (capacity) or #1/#6 (deploy/shared dep) |
| Caller errors but DLP clean | `_dp_mtnsproxy` "ASP client for DLP scan timed out" present; `_nsdlp_svr` errors absent | #13 (upstream) |

**Decisive test:** pick one failing tenant, search its `tenantid` across all hosts in the pop for the same window. Distribution across `_sourceHost` separates tenant-config (every host) from host-local (one host) from fleet-wide (all hosts, many tenants) in one query.

**Known patterns (differential diagnosis):**
- **Host stall + HAProxy flush burst:** ~3-min window, one host's errors spike then a 10–50× burst of queued requests flushes to the recovering host; self-recovers in ~10 min. Rule out: tenant config (same tenants succeed on peers), capacity (peers clean), crash (no crash-dump lines). The 100%-fail "tenants" are nearly all low-volume (reqs < 10) — alert grouping `(by host, tenantid)` amplifies these; treat as noise floor unless many cluster on one host. Don't act unless recurring.
- **Broken *profile* (not whole-tenant config):** a tenant has multiple DLP profiles; only scans invoking the broken one fail, so fail rate may be a few % to 100% — **don't wait for 100%**. Climbs over hours as more scans match the broken profile's applicability. Different pops may show 0 failures (their traffic mix doesn't hit that profile). `profile_load_failure` strings track 1:1 with `failed_scans` for the tenant. Act at a few %.

**Gotchas:**
- **dlpsvc is C++ — no GC pauses.** For unexplained stalls, look for blocked threads, stuck downstream-dep sub-requests (redis/OCR/classifier/config-fetch), fd exhaustion, IO stalls — not "GC".
- **OCR failures don't fail scans** (best-effort). Correlation ≠ causation; don't pivot to OCR unless image-attachment OCR is the reported symptom.
- **`failed_scans` counter line has no `client` field** — for per-client attribution, grep the scan's `"All Scan Info"` line (carries `client=`) or the rqid trace.
- **Zero `IncidentReport.cpp` lines (neither success nor fail) for the alerting tenant/window = detection-latency, not data loss.** Scans are timing out before ever reaching the incident/forensic-post stage — don't pivot to playbook #9/#11 until post-attempt lines actually start appearing.
- **Even distribution across all hosts in a pool = do not drain any single host.** If a tenant-driven timeout/error count is spread roughly evenly (not concentrated on one host), draining a host reduces aggregate capacity and worsens the alert — the opposite corrective action from the single-bad-host case.
- **If the failing/killed scans' file size isn't logged** (killed before "All Scan Info" emits it), bucket the *successful* scans' sizes for the same tenant/host/window instead — the ceiling of that distribution (e.g. successes cap at 33.5MB) gives an actionable size threshold even without directly measuring the failures.

### Incident-post surface (rows 9, 11)

**7-mode classifier (Sumo-grounded):**

| Mode | Sumo signal | Class |
|---|---|---|
| 1 | `IncidentReport.cpp` "Retrying incident post ... attempt: N" then "succeeded" (num_retries≥1, status=ok) | Transient — retries absorb |
| 2 | "incident file upload failed ... final=true" + GEF fallback fires (no succeeded line) | Direct path dead — real loss |
| 3 | GEF fallback fires but forensic payloads absent (UI `forensic_status="Upload Failure"`) | By design — metadata-only, forensic lost |
| 4 | Incidents delayed in UI (Kafka `ims-incident-cg` lag — needs Prism to confirm; Sumo can't see lag directly) | GEF-path consumer bottleneck |
| 5 | `error=resp_400` on `/forensic` + IMS `dlp_incidents.py:235` `ClientDisconnected '400: Bad Request'` + `Missing event_file` + uwsgi `Nmsecs` ≥120000 | Transport truncation (NOT bad request) |
| 6 | IMS logs `FolderFullException on forensic upload` or ims-ceph errors | Forensic storage backend / customer folder full |
| 7 | `forensic_status="Upload Failure"` despite healthy direct path (no resp_400/timeout in DLP logs) | DLP forensics buffer at 8 GB cap — backpressure, usually secondary to 1/2/5 or slow scan (pivot to scan) |

**Log patterns (file:line fingerprints):**
- `IncidentReport.cpp:02442` — attempt summary: `res=1 statuscode=400 url=... attempt=N maxAttempts=3 haveRetries=true haveMoreEndpoints=true isFinal=false`
- `IncidentReport.cpp:02495` — failure: `error=resp_400 url=... size=... retry=0 final=false endpoint='LBaaS'`
- `IncidentReport.cpp:02579` — retry schedule: `Retrying incident post to LBaaS at ..., attempt: 2, after=30000ms`
- `IncidentReport.cpp:02502` — success: `incident file upload succeeded. url=... retry=2`
- IMS `dlp_incidents.py:1180` — `Processing forensic for incident <id> tenant <tid>`
- IMS `dlp_incidents.py:203-205` — request args + `Content-Length` + `files keys: ['event','forensics-zip','original-data-zip']`
- IMS `dlp_incidents.py:235` — `Error getting event for tenant: <tid>, incident_id: <id>. Error: <ClientDisconnected '400: Bad Request'>` + `Missing event_file` (mode 5)
- uwsgi access log — `POST /forensic/<tid>/<id>?source=dlp&try_count=N => generated 78 bytes in Nmsecs (HTTP/1.1 400)` — `Nmsecs` ≥120000 = truncation
- `FolderFullException on forensic upload` (mode 6 — customer SaaS folder full, non-retryable)

**Endpoint resolution** (from `/opt/ns/cfg/ims-apisvc-service.json` per-DC): DLP VM intra-pop → LBaaS intra VIP; inter-pop (tenant home_pop ≠ this MP) → LBaaS inter VIP; DLP k8s same cluster → `ims-apisvc.ims.svc:8000`; k8s different cluster → LBaaS inter VIP; LBaaS disabled/fails → `apiconnectorsvclbvip-int.<pop>:8000`. Explains which `ims_server` a tenant's posts hit.

**Gotchas:**
- **`is_final` / `isFinal` = "incident fully done", NOT "last retry".** A forensic-post success with `isFinal=false` means another part (original_file) is still pending. Don't read `isFinal=false` as "more retries coming".
- **`num_retries` appears on success AND fail rows.** `num_retries=3, status=ok` = recovered but transient — still investigate. `num_retries=5, status=fail` = all 6 attempts failed → GEF.
- **`status_string` may be HTTP code OR curl error string.** HTTP: 200/202/400/500/503/504/521. Curl (transport, no IMS reply): `timeout`, `Could not connect to server`, `Could not resolve hostname`, `Failure when receiving data from the peer`, `Server returned nothing`, `Failed sending data`. Curl errors → network/DNS/LB, not IMS app.
- **`521` = "Kafka-only failure, incident persisted anyway"** (IMS spec). DLP retries 521 → possible duplicate incidents. Don't panic unless persistent.
- **`/incident` (metadata only) vs `/forensic` (metadata + forensic zips)** — different endpoints, different payloads. Split on URL path in queries; a pop can be healthy on one and broken on the other.
- **DLP is bit-identical on retries** — same trid, rqid, content, Content-Length; only `try_count` in URL and retry counter change. So 400 on attempt 1 + 200 on attempt 3 with same payload = transport, not content.
- **`haveMoreEndpoints=true` ≠ "next attempt switches endpoint".** DLP exhausts LBaaS (3 attempts) before switching to apiconnectorsvclb (3 more). `attempt=3 maxAttempts=3 haveMoreEndpoints=true` = LBaaS exhausted, attempt 4 = APIC.
- **IMS runs Python** (`dlp_incidents.py`) — GC pauses ARE possible (unlike dlpsvc C++). "Slow IMS" can mean Python GC, Celery queue backup, or uwsgi worker saturation.

### EG surface (row 14 — the k8s `dlp-event-generator`, distinct from legacy ZMQ EG in #9)

**Sumo log patterns (primary path without Prism):**
- EG logs trace a `request_id` end-to-end. **TEMPORARY gotcha:** EG logs are in the Sumo **infrequent tier** — every query MUST prefix `_index="dlp_<pop>_debug"` (lowercase pop, `-`→`_`, e.g. SG-SIN2 → `dlp_sg_sin2_debug`) alongside `_sourceCategory=*dlp-event-generator*`, or it returns nothing.
- **Lifecycle RECOVERED:** `Processing incident...` → `Send attempt failed` → `Exhausted retries for endpoint` → `Posting forensic to IMS (Pop-List)` → `forensic successfully posted to IMS (Pop-List)`.
- **Lifecycle LOST:** `Send attempt failed` → `Exhausted retries` (×endpoints) → `incident drop: all IMS endpoints exhausted` = DATA LOST.
- **Error patterns:** `context deadline exceeded` (5-min `send_timeout`), `connection reset by peer` (remote killed mid-transfer), `connection refused` (endpoint down), `i/o timeout` (network).
- **Bulk recovered-vs-lost per `request_id`:** parse `"request_id":"<id>"` + `"msg":"<...>"`; `msg` matches `*successfully posted*` → RECOVERED, `*drop*` → LOST. `RECOVERED>0` and `LOST` null = fallback worked, no data loss.

**Architecture:** EG = k8s Deployment (`ns=dlp-event-generator`, 2–5 replicas/pop, real pods `dlp-event-generator-<hash>-<id>`). Pulls incidents/alerts off a Redis stream, downloads forensic blobs from S3, decrypts via Vault, posts multipart to IMS with own fan-out: `ims-apisvc.ims:8000` (local) → `imsapisvclbaasvip-int.<region>` → `apiconnectorsvclbvip-int.<region>`. On full exhaustion: own GEF fallback (incident metadata only — forensic binary lost, same semantic as DLP-side GEF). `dlp-event-generator-test-connection` is a Helm test hook → `pod_not_running` on only it = FALSE POSITIVE (already in row 14 / recurring alerts).

- OCR path is separate: SMTP-proxy-sent OCR requests show as `client="service-smtp-proxy"` on the OCR side. SMTP-DLP slowness may be OCR-driven only if image attachments are in the path.

## Escalation notes

- Threshold-crossing prod issues get an **IMF/EIMF**, not a bare OPS ticket (mandated after the BOG1 incident).
- FedRAMP/PBMM stacks are not visible from India Prism — route to US on-call.
- **China (PRC) dataplane** uses separate Prism datasources with a `-prc` suffix (`vm-dp-prc`, `vm-dlp-prc`, `vm-haproxy-prc`, `vm-system-prc`, `vm-k8s-prc`) reached via `teleport.cnnetskope.io` cluster `china-prod` — the global Prism on `aws-api.prism.netskope.io` does NOT have PRC datasources. MP-side datasources (`vm-apic`, `vm-phx`, `vm-kafka`, `vm-eventforwarder`) are shared. Metric names/labels are identical across PRC and global; only the datasource routing differs.
- CM-gated changes to prod (vendor debug env vars, config edits, EDK-chunking flag enables) need sign-off from Emanoel/Niranjan/Abhiram first — don't self-approve.
- **The paging/escalation system itself can be the incident** (EIMF-569, 2026-07): OpsGenie's escalation path paged the entire on-call team, not just primary, for a P1 whose occurrence count was below the documented EIMF threshold. If page volume looks disproportionate to the underlying signal, check the escalation-path config before assuming the underlying alert is as severe as the paging volume suggests.
